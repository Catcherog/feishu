项目根目录中的 `PROJECT_GUIDE.md` 是长期事实源。开始任何工作前必须完整阅读，并严格遵循其中的：

若真实代码、飞书结构和指南不一致，不要自行选择答案。请记录差异，输出审计结果，等待确认。

1. 飞书继续作为当前业务事实主库。
2. 将模特、化妆师、场地、服装、修图师等资源合并为统一 `Resource` 表，通过 `resource_type` 和视图区分。

9. 首期范围只聚焦客户、项目、模特、化妆师、策划案和必要任务。
10. 禁止提交真实 App Secret、Token、API Key、客户隐私或内部资源 ID。

## 你现在只执行 Phase 0，不进入 Phase 1

### Phase 0 目标

完成当前状态审计与可恢复备份，不修改线上业务结构。

### 必须执行的工作

#### 1. 读取与盘点

检查并列出：

- 项目目录结构
- 所有与飞书相关的配置和脚本
- 当前 Base、Table、Field、View 的真实资源映射
- 12 条旧自动化的真实配置
- Expo APP 当前读取和写入的表
- 数据同步脚本实际支持的表
- 所有历史报告中互相冲突的信息

不得仅根据旧 README 输出结论。

#### 2. 安全备份

生成：

- 当前 Base 结构导出
- 每张表的数据导出或可恢复快照
- 自动化规则配置备份
- 视图和字段映射备份
- Git 当前 Commit、工作树状态和文件清单

备份文件不得包含 Secret。真实资源 ID 放在本地忽略文件中，公开文档只使用别名。

#### 3. 真实使用度审计

对每张表输出：

- 记录数
- 最近 30 天新增数
- 最近 30 天更新数
- 最后更新时间
- 是否被 APP 读取
- 是否被脚本读取/写入
- 是否被自动化引用
- 是否有真实人工使用
- 建议状态：保留、合并、迁移、归档、待确认

重点核查客户、项目、模特、化妆师、服装、场地、修图师、知识和发布相关表。

#### 4. 自动化真实运行审计

对于每条旧自动化，不接受“已启用”作为运行证据。输出：

- 是否启用
- 最近执行时间
- 最近执行次数
- 成功次数
- 失败次数
- 是否造成重复记录
- 是否存在 Loop 上限遗漏
- 是否可找到真实输入和真实输出记录
- 是否仍有业务价值
- 建议：关闭、保留原生、迁移 Serverless、删除前归档

若 API 无法读取执行日志，明确写出限制，并设计人工验证步骤。

#### 5. 生成差异报告

比较：

- 历史文档声称的结构
- 当前真实 Base 结构
- 当前 APP 代码映射
- 当前同步脚本映射

输出所有不一致，禁止静默修复。
